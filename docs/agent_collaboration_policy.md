# Go2W 跨楼层自主导航项目协作规约

## 1. 规约目标
本规约用于统一本项目的人机协作方式、多模型分工方式、代码实现边界与评审流程，确保开发过程满足以下原则：
* 先闭环，再升维
* 单任务、单责任人、单验收标准
* 研究与实现解耦
* 评审与实现分离
* 严禁多写手并行改动同一任务

本规约适用于本项目全部阶段，包括仿真、控制、SLAM、拓扑路由、楼梯状态机、高程地形感知等模块。

## 2. 协作角色与职责
### 2.1 人类总控（项目负责人）
**职责：**
* 定义阶段目标与任务边界
* 选择当前唯一实现任务
* 审批实现方案与验收标准
* 决定是否合并、返工或冻结任务
* 负责最终技术路线裁决

**说明：** 项目负责人是唯一调度者与最终决策者，不是被动执行者。

### 2.2 GPT + Gemini 一号：技术研究与路线确认组
**职责：**
* 搜集官方文档、GitHub 仓库、论文与开源实现资料
* 形成候选技术路线
* 对方案可行性进行交叉验证
* 输出任务前置判断，而不直接改代码

**边界：** 可以提出架构建议，不参与主仓库代码实现，不直接替代 Cursor 写代码。

### 2.3 Cursor + Codex：唯一主写手
**职责：**
* 按已批准任务单进行代码实现
* 构造文件、修改代码、运行命令、完成最小验证
* 输出改动摘要与执行结果

**边界：** 是唯一允许直接撰写主仓库代码的实现者。不得擅自扩展任务范围，不得在未定义验收标准前开始大规模改动。

### 2.4 GPT 审计席：代码改动审计
**职责：**
* 基于 GitHub 仓库内容进行代码改动审查
* 检查目录结构、接口一致性、命名规范、架构边界
* 判断实现是否偏离当前阶段目标

**边界：** 只做审计，不做主代码编写。以“是否符合任务单和阶段蓝图”为主要判断标准。

### 2.5 Gemini 二号：反证与异常挑战席
**职责：**
* 做边界场景质疑
* 检查潜在 bug、失败模式与恢复路径
* 识别异常隔离问题、耦合问题、隐含风险

**边界：** 不直接改主仓库代码。重点关注“此实现在哪些条件下会失败”。

## 3. 核心协作原则
### 3.1 唯一主写手原则
任何时刻，只有 Cursor + Codex 可以执行主仓库代码撰写。
**禁止：** GPT 直接代写主代码；Gemini 直接代写主代码；多模型同时改同一任务代码。

### 3.2 并行限制原则
并行仅允许发生在以下两个环节：研究、评审。
**禁止：** 多实现者并行改同一任务；在未完成当前阶段验收前并行推进多个强依赖实现任务。

### 3.3 单任务单窗口原则
每个窗口、每次会话、每个实现任务，只处理一个明确任务。
**禁止：** 一个窗口同时推进仿真、SLAM、导航、状态机多个主题；以“大而全 prompt”一次性推进多个阶段目标。

### 3.4 评审与实现分离原则
实现者不得充当最终验收者。代码实现完成后，必须经过：GPT 仓库审计、Gemini 二号反证审查、人类总控最终裁决。

### 3.5 跨模型信息同步原则（Single Source of Truth）
为降低跨模型调度的上下文丢失与错位风险，系统必须维护一个共享的单一事实源。
**要求：**
* 在 `docs/architecture_state.md` 中记录最新的架构状态、接口契约和阶段里程碑。
* 每次阶段任务完成或发生重大架构调整时，必须更新此文档。
* 所有 Agent（特别是 Cursor 与 GPT 审计席）在开启新对话任务时，需优先读取该文档以对齐最新上下文。

## 4. Cursor/Codex 标准任务单格式
每次发给 Cursor/Codex 的实现任务，必须固定包含以下 6 项：
1. 任务目标
2. 当前阶段
3. 可改文件范围
4. 不可改文件范围
5. 必跑命令
6. 验收标准（Definition of Done）

未满足以上 6 项时，不得进入正式实现。

## 5. 统一输出格式
所有参与协作的 agent 原则上统一按以下格式输出，保证信息可交接、可审计、可复核：
* 结论
* 依据
* 假设
* 风险
* 待验证项
* 下一步建议

**要求：** 不输出无结构长文；不混淆“已证实事实”和“推测判断”；不省略风险与待验证项。

## 6. Git 与任务执行规范
### 6.1 特性分支工作流与 PR 审查 (Feature Branching & PR)
绝对禁止直接在 `main` 分支上进行开发提交。`main` 分支必须永远保持稳定与可用。所有开发严格采用特性分支工作流：
1. **切分战场**：领取任务时，基于 `main` 切出新分支（如 `feat/phase0-setup`）。
2. **独立开发**：Cursor 在此特性分支上完成代码撰写与小步提交。
3. **提交审核**：推送到 GitHub 并向 `main` 发起 Pull Request (PR)。
4. **自动审计**：交由 GPT 审计席对该 PR 进行全自动代码审查。
5. **合并主干**：仅在审查通过并由人类总控点击 Merge 后，代码才允许合入 `main`。禁止在同一分支上并行推进多个不相关任务。

### 6.2 先验收标准，后实现
必须先定义 DoD，再允许代码实现。禁止“先写代码、后想怎么验收”。

### 6.3 小步提交
每个任务采用小步提交策略：
* 一次只完成一个逻辑目标
* 一次提交只对应一个明确意图
* 避免大规模无边界改动

## 7. 任务升级与冻结规则
当出现以下情况时，应暂停当前实现，回到研究/评审阶段：
* 多个 agent 对技术路线结论严重冲突
* 当前实现超出阶段目标
* 出现跨模块级联修改
* 验收标准无法客观执行
* 代码实现引入不可控技术债

必要时，项目负责人有权冻结当前任务，重新定义边界。

## 8. 本规约的最终解释权
本规约由项目负责人维护并拥有最终解释权。当实际工程需求与本规约冲突时，以“降低耦合、控制范围、确保闭环”为优先准则进行调整。
